Agent Team Factory

How to Use and Improve the Factory

This tutorial walks through local setup, deterministic workflows, permissions, memory, evaluations, the opt-in LLM adapter, and safe ways to extend the factory. The default path is deterministic and reviewable; real LLM calls are strictly opt-in.

Quick start

Run these commands from the repository root:

cd /Users/cosdis/Desktop/AI_builder/AI_Team_building
~/.venvs/myenv/bin/python -m pip install -e '.[dev]'
~/.venvs/myenv/bin/python -m pytest
scripts/ci_regression.sh

The regression script validates specs, checks golden snapshots, runs deterministic mock evaluations, compares traces, exercises run logs, runs a deterministic LLM adapter smoke test, and checks the local API dispatcher.

Repository tour

  • team_specs/*.yaml — user-authored team definitions.
  • src/team_factory/specs/ — Pydantic schema and YAML loader.
  • src/team_factory/orchestration/ — deterministic mock runtime.
  • src/team_factory/tools/ — permission checks, no execution.
  • src/team_factory/memory/ — SQLite local memory foundation.
  • src/team_factory/evaluation/ — deterministic scenario reports.
  • src/team_factory/observability/ — traces, run logs, audit logs.
  • src/team_factory/api/ — local API skeleton.
  • src/team_factory/llm/ — deterministic and opt-in LLM adapters.
  • tests/golden_snapshots/ — checked-in regression baselines.

1. Validate team specs

Start every change by validating the YAML specs.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py validate team_specs/*.yaml

Validation catches malformed agent definitions, workflow references, safety policy issues, and high-impact tool approval gaps before runtime code is involved.

2. Inspect workflow order

Use workflow order checks to understand which agents will run in mock mode.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py workflow-order \
  team_specs/travel_planning_team.yaml \
  --workflow-id plan_trip

3. Run deterministic mock workflows

Mock runs are the safest way to debug specs and orchestration shape.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-mock \
  team_specs/travel_planning_team.yaml \
  "Plan a weekend city trip." \
  --workflow-id plan_trip

For machine-readable output, add --json.

4. Record and compare traces

Trace snapshots make deterministic workflow drift visible.

mkdir -p examples/artifacts/tutorial
~/.venvs/myenv/bin/python scripts/team_factory_cli.py trace-snapshot \
  team_specs/travel_planning_team.yaml \
  "Plan a weekend city trip." \
  --workflow-id plan_trip \
  --out examples/artifacts/tutorial/travel_trace.json

~/.venvs/myenv/bin/python scripts/team_factory_cli.py trace-compare \
  tests/golden_snapshots/travel_planning_team__plan_trip__weekend_city_trip.json \
  examples/artifacts/tutorial/travel_trace.json

5. Use golden snapshots intentionally

Check golden snapshots before merging runtime-affecting changes:

~/.venvs/myenv/bin/python scripts/team_factory_cli.py golden-check team_specs/*.yaml

Only update baselines when the change is expected and reviewed. The update command requires explicit approval: 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py golden-update team_specs/*.yaml --approve

6. Run deterministic evaluations

Generate scenario reports for the example teams:

~/.venvs/myenv/bin/python scripts/team_factory_cli.py eval \
  team_specs/scientific_discovery_team.yaml \
  team_specs/trading_strategy_research_team.yaml \
  team_specs/travel_planning_team.yaml \
  --out-dir examples/artifacts/evaluation_reports

Reports are Markdown files under the selected output directory.

7. Check tool permissions without executing tools

The tool registry authorizes or rejects proposed calls. It does not execute them. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py tool-check \
  team_specs/trading_strategy_research_team.yaml \
  --agent-id risk_reviewer \
  --tool-id backtest_engine \
  --purpose "Review strategy risk" \
  --permission read_market_data

If a tool is high-impact or critical, specs and authorization flows must require human approval before future execution support is considered.

8. Manage local memory

The memory store is currently standalone and useful for testing retention/redaction.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-put \
  --db examples/artifacts/tutorial/memory.sqlite3 \
  --category project \
  --key tutorial_note \
  --value-json '{"note":"remember this local preference"}'

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-list \
  --db examples/artifacts/tutorial/memory.sqlite3

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-get \
  --db examples/artifacts/tutorial/memory.sqlite3 \
  --category project \
  --key tutorial_note

9. Generate text with the LLM adapter layer

The factory now uses gpt-5.5-codex as the default model identifier with medium reasoning effort. The provider is still deterministic unless real LLM use is explicitly enabled. Override the model/reasoning defaults with --model, --reasoning-effort, TEAM_FACTORY_DEFAULT_LLM_MODEL, or TEAM_FACTORY_DEFAULT_LLM_REASONING_EFFORT.

Deterministic adapter using the Codex model identifier, safe for CI and local regression:

~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --instructions "Be concise."

Real OpenAI Responses adapter, strict opt-in and still no tools:

export TEAM_FACTORY_ENABLE_REAL_LLM=1
export OPENAI_API_KEY=...
~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --provider openai_responses \
  --model gpt-5.5-codex \
  --reasoning-effort medium \
  --enable-real-llm

The real adapter sends tools=[], tool_choice="none", and parallel_tool_calls=false, with reasoning.effort="medium" by default. It is not wired into agent runtime orchestration yet.

10. Try the local API skeleton

Start the local API server:

~/.venvs/myenv/bin/python scripts/team_factory_api.py --host 127.0.0.1 --port 8765

In another terminal:

curl http://127.0.0.1:8765/health

curl -X POST http://127.0.0.1:8765/specs/validate \
  -H 'Content-Type: application/json' \
  -d '{"spec_path":"team_specs/travel_planning_team.yaml"}'

The API is a local development skeleton and should not be exposed as production service.

How to improve the factory safely

  1. Add or edit a team spec first. Define agents, workflows, tools, safety policy, and evaluation scenarios in YAML.
  2. Validate before runtime changes. Run spec validation and workflow-order checks after every spec edit.
  3. Keep deterministic mocks current. Add or update tests before introducing nondeterministic behavior.
  4. Use golden snapshots intentionally. Check snapshots by default; update them only with --approve after review.
  5. Do not add tool execution casually. Any real tool execution phase should include approvals, audit logs, risk levels, and narrow allowlists.
  6. Keep real LLM usage opt-in. New real-provider paths should retain deterministic fallbacks and stay out of CI by default.
  7. Document each phase. Add a focused docs/phase_*.md file and update the progress dashboard.

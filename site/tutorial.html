Agent Team Factory

Use the Factory First, Then Improve It

Start by running the factory exactly as it works today: validate a team spec, inspect workflow order, run deterministic mock workflows, evaluate results, and try the safe LLM adapter. After that, use the second half of this tutorial to improve the factory without breaking deterministic regression or safety gates.

What you will learn

Part 1 — Use it

  1. Install and verify the local package.
  2. Validate existing team YAML specs.
  3. Run mock workflows and inspect outputs.
  4. Use traces, golden snapshots, evaluations, memory, and local API calls.
  5. Try the deterministic LLM adapter and understand strict real-LLM opt-in.

Part 2 — Improve it

  1. Edit or add team specs safely.
  2. Add scenarios, tests, docs, and website updates.
  3. Extend runtime support while preserving deterministic mocks.
  4. Plan real LLM/tool execution only behind explicit safety gates.
Part 1

Use the factory today

Follow these steps in order. They teach the current factory without requiring real LLM calls, external tools, or production services.

0. Start from the repo root

Run these commands once in a terminal:

cd /Users/cosdis/Desktop/AI_builder/AI_Team_building
~/.venvs/myenv/bin/python -m pip install -e '.[dev]'
~/.venvs/myenv/bin/python -m pytest

If this passes, the Python package, tests, and local development environment are ready.

1. Validate the team specs

Specs are the main user-facing input. Validate them before running or changing anything. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py validate team_specs/*.yaml

You should see each spec marked OK.

2. Inspect a workflow before running it

Workflow order shows which agents the deterministic mock runtime will execute. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py workflow-order \
  team_specs/travel_planning_team.yaml \
  --workflow-id plan_trip

Use this when learning a team or checking that a YAML workflow references the expected agents.

3. Run a deterministic mock workflow

Mock runs are the safest way to understand team behavior because they are repeatable and do not call real LLMs or tools. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-mock \
  team_specs/travel_planning_team.yaml \
  "Plan a weekend city trip." \
  --workflow-id plan_trip

For machine-readable output, add --json.

4. Save a trace and compare it

Trace snapshots turn a run into a deterministic JSON artifact. This is how you detect accidental workflow drift. 

mkdir -p examples/artifacts/tutorial
~/.venvs/myenv/bin/python scripts/team_factory_cli.py trace-snapshot \
  team_specs/travel_planning_team.yaml \
  "Plan a weekend city trip." \
  --workflow-id plan_trip \
  --out examples/artifacts/tutorial/travel_trace.json

~/.venvs/myenv/bin/python scripts/team_factory_cli.py trace-compare \
  tests/golden_snapshots/travel_planning_team__plan_trip__weekend_city_trip.json \
  examples/artifacts/tutorial/travel_trace.json

If the comparison reports a difference, inspect whether the change was intentional.

5. Check golden snapshots

Golden snapshots are checked-in baselines for supported example workflows.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py golden-check team_specs/*.yaml

Only update baselines when the behavior change is expected and reviewed: 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py golden-update team_specs/*.yaml --approve

6. Run deterministic evaluations

Evaluations generate Markdown reports for each example team.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py eval \
  team_specs/scientific_discovery_team.yaml \
  team_specs/trading_strategy_research_team.yaml \
  team_specs/travel_planning_team.yaml \
  --out-dir examples/artifacts/evaluation_reports

Open the generated Markdown files under examples/artifacts/evaluation_reports.

7. Check tool permissions without executing tools

The tool registry authorizes or rejects proposed calls. It does not execute them. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py tool-check \
  team_specs/trading_strategy_research_team.yaml \
  --agent-id risk_reviewer \
  --tool-id backtest_engine \
  --purpose "Review strategy risk" \
  --permission read_market_data

Use this to understand permission rules before any future real tool execution phase.

8. Try local memory

Memory is currently standalone. Use it to test storage, lookup, redaction, and retention. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-put \
  --db examples/artifacts/tutorial/memory.sqlite3 \
  --category project \
  --key tutorial_note \
  --value-json '{"note":"remember this local preference"}'

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-list \
  --db examples/artifacts/tutorial/memory.sqlite3

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-get \
  --db examples/artifacts/tutorial/memory.sqlite3 \
  --category project \
  --key tutorial_note

9. Try the LLM adapter safely

The default model setting is gpt-5.5-codex with medium reasoning effort, but the default provider remains deterministic. That means this first command is safe for CI and local regression. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --instructions "Be concise." \
  --json

Real OpenAI Responses calls require explicit opt-in and still send no tools: 

export TEAM_FACTORY_ENABLE_REAL_LLM=1
export OPENAI_API_KEY=...
~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --provider openai_responses \
  --model gpt-5.5-codex \
  --reasoning-effort medium \
  --enable-real-llm

The real adapter sends tools=[], tool_choice="none", parallel_tool_calls=false, and reasoning.effort="medium". It is not wired into agent runtime orchestration yet.

10. Try the local API skeleton

Start the local API server:

~/.venvs/myenv/bin/python scripts/team_factory_api.py --host 127.0.0.1 --port 8765

In another terminal:

curl http://127.0.0.1:8765/health

curl -X POST http://127.0.0.1:8765/specs/validate \
  -H 'Content-Type: application/json' \
  -d '{"spec_path":"team_specs/travel_planning_team.yaml"}'

The API is a local development skeleton and should not be exposed as a production service.

11. Run the full local regression suite

Use this before committing any meaningful change:

scripts/ci_regression.sh
scripts/release_check.sh

These commands exercise tests, linting, spec validation, golden snapshots, deterministic evaluations, trace comparison, run logs, the deterministic LLM adapter, and the API dispatcher.

Part 2

Improve the factory after you can use it

Once the usage flow above is familiar, improve the factory in small safe increments. Every improvement should preserve deterministic tests and explicit safety gates.

Improvement loop

  1. Edit one thing. Start with a team spec, scenario, test, or small runtime change.
  2. Validate specs. Run team_factory_cli.py validate team_specs/*.yaml.
  3. Run a mock workflow. Confirm behavior is deterministic and understandable.
  4. Check traces/goldens. Use golden-check; update only with --approve.
  5. Run tests and CI regression. Use pytest, ruff, and scripts/ci_regression.sh.
  6. Document the change. Update docs and this website when behavior changes.

Where to make improvements

  • team_specs/*.yaml — define or revise teams, agents, workflows, tools, and scenarios.
  • src/team_factory/specs/ — evolve the schema and validation rules.
  • src/team_factory/orchestration/ — add deterministic workflow runtime support.
  • src/team_factory/tools/ — strengthen permission and approval logic.
  • src/team_factory/memory/ — improve retention, redaction, and future runtime integration.
  • src/team_factory/evaluation/ — add scenarios and report fields.
  • src/team_factory/observability/ — improve traces, run logs, and audit records.
  • src/team_factory/api/ — add local endpoints for developer workflows.
  • src/team_factory/llm/ — expand adapters while keeping strict opt-in.
  • site/ and docs/ — keep users and reviewers oriented.

Easy improvements

  • Add a new evaluation scenario to an existing team spec.
  • Add a new example team YAML and validation tests.
  • Add clearer CLI examples to this tutorial.
  • Add more golden snapshots after deterministic workflows are supported.

Medium improvements

  • Add another mock-compatible workflow pattern.
  • Improve audit/run-log metadata for reproducibility.
  • Add API routes that wrap existing CLI-safe workflows.
  • Add memory retrieval to deterministic mock outputs without making real calls.

Advanced improvements

  • Add a guarded LLM-backed single-agent smoke workflow excluded from CI by default.
  • Add streaming support for real providers behind explicit opt-in.
  • Design real tool execution with approvals, allowlists, audit logs, and dry-run mode first.
  • Plan LangGraph integration only after deterministic behavior and safety gates are stable.

Safety rules for future improvements

  • Default behavior should remain deterministic unless the user explicitly opts into real providers.
  • Real LLM usage must stay behind TEAM_FACTORY_ENABLE_REAL_LLM=1 and CLI/config opt-in.
  • No real tool execution should be added without approval gates, risk levels, and audit logs.
  • Every runtime change should have tests and, when applicable, golden snapshot coverage.
  • Docs and the local website should describe both how to use the feature and how to review it.

Recommended next build

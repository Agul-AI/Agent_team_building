Agent Team Factory

Use the Factory First, Then Improve It

Start by running the factory exactly as it works today: validate a team spec, inspect workflow order, run deterministic mock workflows, evaluate results, and try the safe LLM adapter. After that, use the second half of this tutorial to improve the factory without breaking deterministic regression or safety gates.

What you will learn

Part 1 — Use it

  1. Install and verify the local package.
  2. Validate existing team YAML specs.
  3. Run mock workflows and inspect outputs.
  4. Use traces, golden snapshots, evaluations, memory, and local API calls.
  5. Try the deterministic LLM adapter and understand strict real-LLM opt-in.

Part 2 — Improve it

  1. Edit or add team specs safely.
  2. Add scenarios, tests, docs, and website updates.
  3. Extend runtime support while preserving deterministic mocks.
  4. Plan real LLM/tool execution only behind explicit safety gates.
Part 1

Use the factory today

Follow these steps in order. They teach the current factory without requiring real LLM calls, external tools, or production services.

Concrete example: long-term trading strategy research

Suppose your goal is: "Build an agent team to help me search for good long-term trading strategies." Start with the existing team_specs/trading_strategy_research_team.yaml example. It is designed for research and simulation only, not financial advice and not live trading.

What this team does

  • Generates strategy hypotheses.
  • Checks market-data and backtest assumptions.
  • Reviews drawdown, costs, slippage, and exposure.
  • Critiques overfitting, leakage, and invalid assumptions.
  • Produces a simulation-only research summary.

What it does not do yet

  • It does not fetch live market data.
  • It does not run real backtests yet.
  • It does not place trades or connect brokerage accounts.
  • It does not produce guaranteed-return claims.

What do you need to do?

  1. Use the built-in trading team first. You do not need to create a new team to get started. Run the existing trading_strategy_research_team in deterministic mock mode.
  2. Put your research question in the task string. Example: Research robust long-term trend-following strategies for simulation only.
  3. Read the output as a process check, not a real trading recommendation. Today the runtime is deterministic and mocked. It helps verify team structure and safety, not live market performance.
  4. Only after that, customize the YAML spec. Copy the trading spec into your own long-term strategy spec and edit agents, metrics, and evaluation scenarios.
  5. Keep it simulation-only. Do not add live trading, brokerage connections, or real tool execution until the later safety-gated phases exist.

The commands below use this trading team so you can learn the factory with your long-term-strategy use case first.

You are ready to set up this trading team. What is next?

In the current foundation, "set up the team" means creating a local, repeatable research workspace around the checked-in YAML spec. It does not mean live market data, real backtests, brokerage connections, or trade execution yet.

  1. Create an artifact folder. Keep setup traces, run logs, and evaluation reports together.
  2. Validate the spec. Confirm the trading team YAML is structurally valid.
  3. Inspect the team order. Confirm the supervisor-worker flow is what you expect.
  4. Run your first mock research task. Put your long-term strategy question in the quoted task string.
  5. Save the run and trace. This gives you a baseline for later improvements.
  6. Run deterministic evaluations. Generate a report that checks the team against its simulation-only scenarios. 
mkdir -p examples/artifacts/strategy_building_team

~/.venvs/myenv/bin/python scripts/team_factory_cli.py validate \
  team_specs/trading_strategy_research_team.yaml

~/.venvs/myenv/bin/python scripts/team_factory_cli.py workflow-order \
  team_specs/trading_strategy_research_team.yaml \
  --workflow-id research_backtest_review

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-mock \
  team_specs/trading_strategy_research_team.yaml \
  "Research robust long-term trend-following strategies for simulation only." \
  --workflow-id research_backtest_review \
  --run-log examples/artifacts/strategy_building_team/runs.jsonl \
  --snapshot-out examples/artifacts/strategy_building_team/initial_trace.json

~/.venvs/myenv/bin/python scripts/team_factory_cli.py eval \
  team_specs/trading_strategy_research_team.yaml \
  --out-dir examples/artifacts/strategy_building_team/evaluation_reports

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-log-list \
  --run-log examples/artifacts/strategy_building_team/runs.jsonl

If these commands pass, the team is set up locally in deterministic mock mode. The next engineering step is to run the guarded LLM smoke command below if you want one real model response while still avoiding tools, brokerage, and trade execution.

What your successful output means

If you see persisted run ... to runs.jsonl, wrote trace snapshot ..., and agent outputs ending with final_synthesizer, the local mock team ran successfully. The output is a deterministic process check: it confirms the agents, workflow, logging, and safety boundaries are wired correctly.

If Python says it can't open file ... scripts/team_factory_cli.py, you are probably in the wrong terminal folder. First run: 

cd /Users/cosdis/Desktop/AI_builder/AI_Team_building

Then rerun the command. The script paths in this tutorial are relative to the repo root.

Add and run a guarded LLM-backed single-agent smoke workflow

The factory now includes a guarded smoke workflow command: run-llm-smoke. It sends exactly one selected agent prompt to the real LLM adapter. You can run it through codex_exec to use your local Codex/ChatGPT sign-in quota without a separate API key. It does not run the full team, does not call tools, does not fetch market data, does not backtest, does not connect brokerage accounts, and does not place trades.

Why this is the next safe step

  • It tests one real LLM response before adding full orchestration.
  • It requires TEAM_FACTORY_ENABLE_REAL_LLM=1.
  • It requires --enable-real-llm.
  • It requires --acknowledge-no-tools.
  • It requires --acknowledge-simulation-only.
  • It writes a safe JSON artifact if you pass --out.

Recommended command: use Codex quota, no API key

export TEAM_FACTORY_ENABLE_REAL_LLM=1

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-llm-smoke \
  team_specs/trading_strategy_research_team.yaml \
  "Research robust long-term trend-following strategies on ETF for simulation only." \
  --agent-id strategy_ideator \
  --provider codex_exec \
  --model gpt-5.5-codex \
  --reasoning-effort medium \
  --enable-real-llm \
  --acknowledge-no-tools \
  --acknowledge-simulation-only \
  --out examples/artifacts/strategy_building_team/llm_smoke_strategy_ideator.json

This command uses the codex CLI already installed with the Codex app. It launches an ephemeral codex exec session in an empty temporary folder with --sandbox read-only, ignored user/project rules, and approval_policy="never". In other words, it spends Codex usage for one text response, but it is still not a tool-execution or trading system. The command can keep the factory default gpt-5.5-codex; internally the codex_exec adapter maps it to the Codex CLI model id gpt-5.5.

Optional command: use OpenAI API credits instead

export TEAM_FACTORY_ENABLE_REAL_LLM=1
export OPENAI_API_KEY=...

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-llm-smoke \
  team_specs/trading_strategy_research_team.yaml \
  "Research robust long-term trend-following strategies on ETF for simulation only." \
  --agent-id strategy_ideator \
  --provider openai_responses \
  --model gpt-5.5-codex \
  --reasoning-effort medium \
  --enable-real-llm \
  --acknowledge-no-tools \
  --acknowledge-simulation-only \
  --out examples/artifacts/strategy_building_team/llm_smoke_strategy_ideator_api.json

Start with strategy_ideator because it is the supervisor/planner in research_backtest_review. Later, you can run separate smoke tests for risk_manager or overfitting_critic, but keep them as separate single-agent checks until full LLM orchestration is deliberately added.

How to inspect the result

cat examples/artifacts/strategy_building_team/llm_smoke_strategy_ideator.json

The artifact records the team id, agent id, task, model, reasoning effort, safety mode, prompt, output text, request id, and token usage. It should show no_tools: true, no_brokerage: true, and no_trading_execution: true.

If the command fails

  • If it says TEAM_FACTORY_ENABLE_REAL_LLM, export the env var exactly as shown.
  • If codex_exec says it cannot find codex, run command -v codex or pass --codex-bin /path/to/codex.
  • If codex_exec says you are not signed in, run codex login and choose ChatGPT sign-in.
  • If openai_responses says missing API key, set OPENAI_API_KEY or use --provider codex_exec instead.
  • If it asks for acknowledgements, add both required acknowledgement flags.
  • If it says agent id not found, choose an agent id from the YAML spec.

Where do I put my task?

Use examples/artifacts/strategy_building_team for generated outputs such as run logs, traces, and reports. The folder name is flexible, but keep it under examples/artifacts/ because that path is ignored by git. Use plural artifacts, not singular artifact.

Put one-off research tasks in the runtime command, not in the YAML file. The YAML defines the team structure; the quoted task string defines what you want this run to research. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-mock \
  team_specs/trading_strategy_research_team.yaml \
  "Research robust long-term trend-following strategies for simulation only." \
  --workflow-id research_backtest_review \
  --run-log examples/artifacts/strategy_building_team/runs.jsonl \
  --snapshot-out examples/artifacts/strategy_building_team/initial_trace.json

Add tasks to trading_strategy_research_team.yaml only when you want a reusable evaluation scenario. In that case, add an item under evaluation.scenarios with an id, an input, and expected_properties. Do not put ad hoc tasks into workflows; workflows describe agent order and stopping criteria. 

evaluation:
  scenarios:
    - id: "long_term_trend_following_research"
      input: "Research robust long-term trend-following strategies for simulation only."
      expected_properties:
        - "states that results are not guaranteed"
        - "includes drawdown, costs, turnover, and slippage"
        - "flags overfitting and data leakage risks"

0. Start from the repo root

Run these commands once in a terminal:

cd /Users/cosdis/Desktop/AI_builder/AI_Team_building
~/.venvs/myenv/bin/python -m pip install -e '.[dev]'
~/.venvs/myenv/bin/python -m pytest

If this passes, the Python package, tests, and local development environment are ready.

1. Validate the team specs

Specs are the main user-facing input. Validate them before running or changing anything. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py validate team_specs/*.yaml

You should see each spec marked OK.

2. Inspect a workflow before running it

Workflow order shows which agents the deterministic mock runtime will execute. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py workflow-order \
  team_specs/trading_strategy_research_team.yaml \
  --workflow-id research_backtest_review

Use this when learning a team or checking that a YAML workflow references the expected agents.

3. Run a deterministic mock workflow

Mock runs are the safest way to understand team behavior because they are repeatable and do not call real LLMs or tools. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-mock \
  team_specs/trading_strategy_research_team.yaml \
  "Research a long-term diversified trend-following strategy for simulation only." \
  --workflow-id research_backtest_review

This is your first practical answer to "what do I need to do?" Put your strategy research question in the quoted task string and run the mock workflow. For machine-readable output, add --json.

4. Save a trace and compare it

Trace snapshots turn a run into a deterministic JSON artifact. This is how you detect accidental workflow drift. 

mkdir -p examples/artifacts/tutorial
~/.venvs/myenv/bin/python scripts/team_factory_cli.py trace-snapshot \
  team_specs/trading_strategy_research_team.yaml \
  "Evaluate a simple moving-average crossover strategy for research and simulation only." \
  --workflow-id research_backtest_review \
  --out examples/artifacts/tutorial/trading_trace.json

~/.venvs/myenv/bin/python scripts/team_factory_cli.py trace-compare \
  tests/golden_snapshots/trading_strategy_research_team__research_backtest_review__moving_average_research.json \
  examples/artifacts/tutorial/trading_trace.json

If the comparison reports a difference, inspect whether the change was intentional.

5. Check golden snapshots

Golden snapshots are checked-in baselines for supported example workflows.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py golden-check team_specs/*.yaml

Only update baselines when the behavior change is expected and reviewed: 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py golden-update team_specs/*.yaml --approve

6. Run deterministic evaluations

Evaluations generate Markdown reports for each example team.

~/.venvs/myenv/bin/python scripts/team_factory_cli.py eval \
  team_specs/scientific_discovery_team.yaml \
  team_specs/trading_strategy_research_team.yaml \
  team_specs/travel_planning_team.yaml \
  --out-dir examples/artifacts/evaluation_reports

Open the generated Markdown files under examples/artifacts/evaluation_reports.

7. Check tool permissions without executing tools

The tool registry authorizes or rejects proposed calls. It does not execute them. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py tool-check \
  team_specs/trading_strategy_research_team.yaml \
  --agent-id backtest_engineer \
  --tool-id backtest_runner \
  --purpose "Run a simulation-only backtest" \
  --permission local_write

Use this to understand permission rules before any future real tool execution phase.

8. Try local memory

Memory is currently standalone. Use it to test storage, lookup, redaction, and retention. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-put \
  --db examples/artifacts/tutorial/memory.sqlite3 \
  --category project \
  --key tutorial_note \
  --value-json '{"note":"remember this local preference"}'

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-list \
  --db examples/artifacts/tutorial/memory.sqlite3

~/.venvs/myenv/bin/python scripts/team_factory_cli.py memory-get \
  --db examples/artifacts/tutorial/memory.sqlite3 \
  --category project \

9. Try the LLM adapter safely

The default model setting is gpt-5.5-codex with medium reasoning effort, but the default provider remains deterministic. That means this first command is safe for CI and local regression. 

~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --instructions "Be concise." \
  --json

If you want to use your Codex quota instead of a separate API key, use codex_exec. This launches the local Codex CLI in a locked-down smoke-test session: 

export TEAM_FACTORY_ENABLE_REAL_LLM=1
~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --provider codex_exec \
  --model gpt-5.5-codex \
  --reasoning-effort medium \
  --enable-real-llm

Real OpenAI Responses API calls are optional. They require explicit opt-in, a separate API key, and still send no tools: 

export TEAM_FACTORY_ENABLE_REAL_LLM=1
export OPENAI_API_KEY=...
~/.venvs/myenv/bin/python scripts/team_factory_cli.py llm-generate \
  "Summarize the platform in one sentence." \
  --provider openai_responses \
  --model gpt-5.5-codex \
  --reasoning-effort medium \
  --enable-real-llm

The real adapter sends tools=[], tool_choice="none", parallel_tool_calls=false, and reasoning.effort="medium". The codex_exec adapter instead uses a locked-down local codex exec process. Neither path is wired into full agent runtime orchestration yet.

Part 2

Improve the factory after you can use it

Once the usage flow above is familiar, improve the factory in small safe increments. Every improvement should preserve deterministic tests and explicit safety gates.

Improvement loop

  1. Edit one thing. Start with a team spec, scenario, test, or small runtime change.
  2. Validate specs. Run team_factory_cli.py validate team_specs/*.yaml.
  3. Run a mock workflow. Confirm behavior is deterministic and understandable.
  4. Check traces/goldens. Use golden-check; update only with --approve.
  5. Run tests and CI regression. Use pytest, ruff, and scripts/ci_regression.sh.
  6. Document the change. Update docs and this website when behavior changes.

Improve the trading example for your own long-term strategy project

After you can run the existing trading team, make a copy and customize it instead of editing the example directly. Keep it research/simulation-only until real data, backtesting, compliance, and approval flows are added. 

cp team_specs/trading_strategy_research_team.yaml \
  team_specs/long_term_strategy_research_team.yaml

Then edit team_specs/long_term_strategy_research_team.yaml:

  • Change team_id to long_term_strategy_research_team.
  • Change name and purpose to long-term, simulation-only strategy research.
  • Add or rename agents such as macro_regime_researcher, portfolio_constructor, or tax_and_turnover_reviewer.
  • Add metrics such as long-term drawdown, turnover, regime robustness, and out-of-sample stability.
  • Add scenarios like "Evaluate a 5-10 year trend-following strategy across equities, bonds, and cash."

A minimal YAML edit might look like this:

team_id: "long_term_strategy_research_team"
name: "Long-Term Strategy Research Team"
purpose: "Research long-term trading strategies for simulation only; do not provide financial advice or place trades."

evaluation:
  metrics:
    - "long_term_drawdown"
    - "turnover"
    - "regime_robustness"
    - "out_of_sample_stability"
    - "overfitting_risk"
  scenarios:
    - id: "multi_asset_long_term_trend_following"
      input: "Evaluate a 5-10 year diversified trend-following strategy across equities, bonds, and cash for simulation only."
      expected_properties:
        - "states that results are not guaranteed"
        - "includes transaction costs, drawdown, and turnover"
        - "flags overfitting and data leakage risks"

After editing, validate and run it:

~/.venvs/myenv/bin/python scripts/team_factory_cli.py validate \
  team_specs/long_term_strategy_research_team.yaml

~/.venvs/myenv/bin/python scripts/team_factory_cli.py workflow-order \
  team_specs/long_term_strategy_research_team.yaml

~/.venvs/myenv/bin/python scripts/team_factory_cli.py run-mock \
  team_specs/long_term_strategy_research_team.yaml \
  "Research robust long-term strategies for simulation only."

If validation fails, fix the YAML references first: every workflow agent id must exist, every enabled high-impact tool must have approval rules, and every scenario should stay clearly simulation-only.

Where to make improvements

  • team_specs/*.yaml — define or revise teams, agents, workflows, tools, and scenarios.
  • src/team_factory/specs/ — evolve the schema and validation rules.
  • src/team_factory/orchestration/ — add deterministic workflow runtime support.
  • src/team_factory/tools/ — strengthen permission and approval logic.
  • src/team_factory/memory/ — improve retention, redaction, and future runtime integration.
  • src/team_factory/evaluation/ — add scenarios and report fields.
  • src/team_factory/observability/ — improve traces, run logs, and audit records.
  • src/team_factory/api/ — add local endpoints for developer workflows.
  • src/team_factory/llm/ — expand adapters while keeping strict opt-in.
  • site/ and docs/ — keep users and reviewers oriented.

Easy improvements

  • Add a new evaluation scenario to an existing team spec.
  • Add a new example team YAML and validation tests.
  • Add clearer CLI examples to this tutorial.
  • Add more golden snapshots after deterministic workflows are supported.

Medium improvements

  • Add another mock-compatible workflow pattern.
  • Improve audit/run-log metadata for reproducibility.
  • Add API routes that wrap existing CLI-safe workflows.
  • Add memory retrieval to deterministic mock outputs without making real calls.

Advanced improvements

  • Add deterministic review/evaluation of guarded LLM smoke artifacts.
  • Add streaming support for real providers behind explicit opt-in.
  • Design real tool execution with approvals, allowlists, audit logs, and dry-run mode first.
  • Plan LangGraph integration only after deterministic behavior and safety gates are stable.

Safety rules for future improvements

  • Default behavior should remain deterministic unless the user explicitly opts into real providers.
  • Real LLM usage must stay behind TEAM_FACTORY_ENABLE_REAL_LLM=1 and CLI/config opt-in.
  • No real tool execution should be added without approval gates, risk levels, and audit logs.
  • Every runtime change should have tests and, when applicable, golden snapshot coverage.
  • Docs and the local website should describe both how to use the feature and how to review it.

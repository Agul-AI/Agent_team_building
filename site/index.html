Agent Team Factory

Plan & Implementation Progress

A local planning dashboard for a spec-driven platform that designs, validates, evaluates, and eventually deploys reusable AI-agent teams. Current completed scope covers Phase 0 through Phase 13 hardening: architecture/docs, specs, deterministic mock orchestration, manifest-only tool permissions, local memory, deterministic reports, local CLI/API flows, JSONL observability, regression trace snapshots, replay run logs, checked-in golden baselines, CI/release checks, a strict opt-in LLM adapter, and a guarded single-agent LLM smoke workflow.

What this platform is

A team factory, not one hard-coded team. Users define new teams using structured YAML specs that describe agents, roles, tools, memory, workflows, evaluation criteria, safety constraints, and deployment intent.

Current implementation

  • Phase 0 docs and repo skeleton are complete.
  • Phase 1 Pydantic models, YAML loading, and validation are complete.
  • Three example team specs are included.
  • A deterministic mock orchestrator for sequential workflows is complete.
  • A manifest-only tool registry and permission layer is complete.
  • A local SQLite memory foundation with retention/redaction is complete.
  • A deterministic evaluation harness with Markdown reports is complete.
  • Scientific, trading, and travel example teams run in deterministic mock mode.
  • Local CLI flows are complete for core developer workflows.
  • Local API skeleton and JSONL observability foundations are complete.
  • Regression trace snapshots and replay-oriented run logs are complete.
  • Checked-in golden snapshots and approval/update workflow are complete.
  • CI regression commands and release checklist are complete.

  • LLM-backed orchestration, actual tool execution, semantic scoring, vector memory, production deployment, and LangGraph integration have intentionally not started.

Architecture

The intended future flow is:

Team Spec YAML -> Validate -> Compile -> Run -> Log -> Evaluate -> Revise -> Deploy

The current implementation includes spec validation, deterministic mock runtime support for the example workflows, manifest-only tool authorization, standalone local memory persistence, deterministic evaluation reports, local CLI flows, a local API skeleton, structured JSONL observability, regression/replay foundations, golden baselines, CI/release checks, and an opt-in LLM adapter layer. LangGraph, real LLM calls, actual tool execution, semantic scoring, vector memory, production deployment, and runtime memory integration are deferred so the system remains simple and inspectable at the foundation.

Safety foundation

  • Every workflow requires hard stopping criteria.
  • Critical enabled tools require approval.
  • High-impact enabled tools require approval.
  • High-impact actions must appear in human-review gates.
